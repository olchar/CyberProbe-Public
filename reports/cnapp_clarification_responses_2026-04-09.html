1. Annotations Pull Request dans Defender for DevOps Exigence 4.5.40

Question : La capacité d'annotation des PR fait-elle partie de l'offre CNAPP incluse, ou dépend-elle de licences additionnelles ?

2. Standards personnalisés mixtes et Secure Score Exigence 4.5.10

Question : Peut-on créer un framework personnalisé combinant contrôles pré-bâtis Microsoft et contrôles personnalisés ? Génère-t-il un score distinct ? S'applique-t-il uniformément au multi-cloud ?

Regroupement mixte

✅ Confirmé. Defender for Cloud permet de créer un custom standard qui regroupe à la fois des recommandations built-in Microsoft ET des recommandations personnalisées (custom recommendations basées sur KQL).

Score de posture

⚠️ Distinction importante :
  • Les custom standards apparaissent dans le tableau de bord Regulatory Compliance avec leur propre pourcentage de conformité (tout comme CIS, PCI-DSS, NIST, etc.).
  • En revanche, seul le MCSB (Microsoft Cloud Security Benchmark) contribue au Secure Score principal.
  • Les custom standards disposent donc d'un score de conformité dédié, mais distinct du Secure Score global.

Accessibilité

Les custom standards sont visibles dans Environment Settings → Security policies et dans le Regulatory Compliance dashboard — au même niveau que les standards par défaut.

Couverture multi-cloud

✅ Confirmé pour Azure, AWS et GCP. La documentation officielle indique : « Create custom recommendations for all clouds (Azure, AWS, and GCP) with a KQL query. » La capacité de regroupement mixte et de scoring s'applique uniformément aux trois clouds via les KQL queries sur Azure Resource Graph.

Prérequis

CapacitéPrérequis
Custom recommendations (KQL)⚠️ Defender CSPM (payant) + Security Admin
Custom recommendations (Azure Policy)✅ Tous les clients (Foundational CSPM)
Custom standardsOwner permissions sur la subscription

3. DSPM et rôle de Microsoft Purview Exigence 4.5.27

Question : Le DSPM est-il entièrement géré depuis Defender for Cloud et inclus dans sa licence, ou exige-t-il le déploiement d'un workspace Purview distinct ?

✅ Le DSPM dans Defender for Cloud est entièrement autonome. Il est géré depuis la console Defender for Cloud, inclus dans la licence Defender CSPM. Il n'exige PAS de déployer un workspace Purview distinct ni d'acquérir des licences Purview additionnelles.

6. Gestion des autres clouds et Azure Policy

Question : Comment la solution fonctionne-t-elle pour les règles des autres clouds si elle utilise Azure Policy ?

🔵 Clarification clé : Azure Policy ne s'applique PAS directement aux ressources AWS/GCP. Le mécanisme diffère selon le cloud.
CloudMoteur d'évaluationMécanisme
Azure Azure Policy (JSON if/then) Évaluation native des ressources ARM. Supporte enforce (deny, DeployIfNotExists) + audit.
AWS / GCP Moteur Defender for Cloud propriétaire Les ressources sont onboardées via connecteurs et projetées dans Azure Resource Graph. Le moteur de recommandations évalue ces ressources via des requêtes ARG (pas via Azure Policy). Assessment only — pas d'enforcement.
Custom recommendations (KQL) Azure Resource Graph Les queries KQL fonctionnent sur les 3 clouds car les ressources AWS/GCP sont projetées dans ARG.

Implications pratiques

  • Les recommandations built-in pour AWS/GCP sont des évaluations propriétaires de Defender for Cloud (pas des Azure Policy définitions)
  • Le MCSB a des contrôles multi-cloud (ex: « Machines should have vulnerability findings resolved » s'applique aux EC2/GCE)
  • Les custom recommendations KQL fonctionnent sur les 3 clouds car ARG indexe les ressources multi-cloud
  • L'enforcement (deny/remediate) est disponible uniquement pour Azure — pour AWS/GCP, Defender for Cloud ne fait que de l'assessment et de la recommandation

7. Exposition publique — Validation externe vs configuration

Question : La détection d'exposition publique se fait-elle par validation d'accès externe, ou uniquement par validation de configuration ?

✅ Les deux. Defender for Cloud utilise une approche hybride combinant analyse de configuration (inside-out) et scan externe actif (outside-in).
MéthodeTypeDescription
Analyse du control-plane Configuration Vérification des IPs publiques, load balancers, NSGs, règles de firewall
Analyse de joignabilité réseau Configuration avancée Analyse des chemins réseau réels : routing, security groups, firewall rules pour déterminer la joignabilité effective
Defender EASM (outside-in) Scan externe actif Intégration avec Defender External Attack Surface Management : crawling et scan actif depuis l'extérieur qui tente de contacter les ressources et vérifie si elles répondent

Ressources évaluées pour l'exposition internet

10. Centralisation des règles sur toute la plateforme

Question : Les règles sont-elles centralisées à un endroit unique pour application sur toute la plateforme (CSPM, PaC, etc.) ?

⚠️ Partiellement. La centralisation maximale est dans Defender for Cloud pour l'évaluation et la recommandation, mais l'enforcement reste spécifique à chaque domaine.
MécanismePortéeCentralisationCapacité
MCSB Azure, AWS, GCP ✅ Centralisé Appliqué automatiquement partout — standard par défaut
Custom standards (Defender for Cloud) Azure, AWS, GCP ✅ Centralisé Assessment/recommandation via KQL multi-cloud
Azure Policy Azure uniquement ✅ Via Management Groups Assessment + enforcement (deny, remediate)
Regulatory compliance dashboard Azure, AWS, GCP ✅ Vue centralisée Vue unifiée de tous les standards (built-in + custom)
DevOps security policies Repos DevOps ⚠️ Séparé Géré dans chaque plateforme DevOps (GitHub, Azure DevOps)
🔵 En résumé : Il n'existe pas de point de contrôle unique qui couvre CSPM + Policy-as-Code + enforcement sur tous les clouds. Defender for Cloud centralise l'évaluation, mais l'enforcement (deny/remediate) reste dans Azure Policy (Azure uniquement) et dans les mécanismes natifs de chaque cloud/pipeline pour AWS, GCP et DevOps.

11. Mode WAAP — Couverture des accès non-Azure

Question : En mode WAAP, comment la solution couvre-t-elle les accès non-Azure ?

⚠️ Azure WAF est un produit Azure-natif. Il ne protège nativement que les applications routées à travers les services réseau Azure. Defender for Cloud n'inclut pas de WAAP cross-cloud intégré.
ScénarioCouvertureMécanisme
Applications hébergées sur Azure (derrière App Gateway ou Front Door) ✅ Couvert Azure WAF natif
Applications non-Azure routées via Azure Front Door ✅ Couvert Front Door peut servir de reverse proxy global devant n'importe quelle origine (AWS, GCP, on-premises). Azure WAF s'applique dans ce cas.
APIs sur Azure API Management ✅ Couvert Defender for APIs + Azure WAF (via Front Door)
Applications purement AWS/GCP sans transit par Azure ❌ NON couvert Nécessite les WAF natifs de ces clouds (AWS WAF, GCP Cloud Armor) ou routage via Azure Front Door
📖 Source : Azure WAF on Application Gateway  |  Protect APIs with WAF on Front Door
